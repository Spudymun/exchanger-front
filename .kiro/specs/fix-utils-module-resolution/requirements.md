# Требования: Исправление разрешения модуля @repo/utils

## Введение

Веб-приложение ExchangeGO не может найти модуль `@repo/utils`, что приводит к ошибке сборки и невозможности запуска приложения. Компонент `SendingCard.tsx` пытается импортировать `useNumericInput` из `@repo/utils`, но модуль не разрешается.

## Требования

### Требование 1: Исправить разрешение модуля @repo/utils

**Пользовательская история:** Как разработчик, я хочу, чтобы все внутренние пакеты монорепозитория корректно разрешались, чтобы приложение могло успешно собираться и запускаться.

#### Критерии приемки

1. КОГДА веб-приложение импортирует функции из `@repo/utils` ТО модуль должен успешно разрешаться
2. КОГДА запускается команда `npm run dev --workspace=web` ТО приложение должно запускаться без ошибок разрешения модулей
3. КОГДА выполняется сборка проекта ТО все зависимости между пакетами должны корректно разрешаться
4. КОГДА используется функция `useNumericInput` из `@repo/utils` ТО она должна быть доступна в компонентах

### Требование 2: Проверить структуру пакета utils

**Пользовательская история:** Как разработчик, я хочу убедиться, что пакет `@repo/utils` правильно настроен и экспортирует необходимые функции.

#### Критерии приемки

1. КОГДА проверяется файл `packages/utils/package.json` ТО он должен содержать корректные точки входа
2. КОГДА проверяется файл `packages/utils/src/index.ts` ТО он должен экспортировать `useNumericInput`
3. КОГДА проверяется структура пакета utils ТО все необходимые файлы должны существовать
4. КОГДА пакет utils импортируется ТО TypeScript должен находить типы

### Требование 3: Обеспечить корректную конфигурацию TypeScript

**Пользовательская история:** Как разработчик, я хочу, чтобы TypeScript корректно разрешал пути к внутренним пакетам монорепозитория.

#### Критерии приемки

1. КОГДА проверяется `tsconfig.json` веб-приложения ТО он должен содержать корректные пути к пакетам
2. КОГДА TypeScript компилирует код ТО он должен находить все внутренние зависимости
3. КОГДА используется автодополнение в IDE ТО оно должно работать для импортов из `@repo/utils`
4. КОГДА выполняется `npm run check-types` ТО не должно быть ошибок типизации

### Требование 4: Проверить зависимости в package.json

**Пользовательская история:** Как разработчик, я хочу убедиться, что все внутренние зависимости правильно объявлены в package.json файлах.

#### Критерии приемки

1. КОГДА проверяется `apps/web/package.json` ТО он должен содержать зависимость `"@repo/utils": "*"`
2. КОГДА проверяется `packages/utils/package.json` ТО он должен иметь корректную конфигурацию экспортов
3. КОГДА выполняется `npm install` ТО все внутренние зависимости должны корректно связываться
4. КОГДА используется синтаксис `"*"` для внутренних пакетов ТО он должен работать корректно (не `workspace:*`)

### Требование 5: Обеспечить работу функции useNumericInput

**Пользовательская история:** Как пользователь формы обмена, я хочу, чтобы поле ввода суммы корректно обрабатывало числовой ввод с учетом особенностей криптовалют.

#### Критерии приемки

1. КОГДА пользователь вводит числа в поле суммы ТО должна работать валидация и форматирование
2. КОГДА используется функция `useNumericInput` ТО она должна возвращать `handleKeyDown` и `formatValue`
3. КОГДА пользователь нажимает клавиши ТО должна работать фильтрация недопустимых символов
4. КОГДА значение форматируется ТО оно должно соответствовать требованиям криптовалюты

## Ограничения

- Решение должно работать в рамках существующей архитектуры монорепозитория
- Не должны нарушаться существующие зависимости других пакетов
- Конфигурация должна быть совместима с Turborepo и Next.js
- Решение должно работать как в режиме разработки, так и в продакшене

## Критерии успеха

- Веб-приложение успешно запускается командой `npm run dev`
- Все импорты из `@repo/utils` работают корректно
- Форма обмена криптовалют функционирует без ошибок
- Сборка проекта проходит без ошибок разрешения модулей
- TypeScript корректно типизирует все импорты
